## 1️⃣ 数据准确性

### 教训 #1: 检查时间戳而非只看日志

| 项目 | 内容 |
|------|------|
| **日期** | 2026-02-09 |
| **场景** | 检查金融学习任务是否成功 |
| **错误** | 只看了日志显示"Running"，但没有验证文件实际修改时间 |
| **后果** | 误以为任务成功执行了 |
| **正确做法** | 验证 `stat filename` 查看文件的实际创建/修改时间 |
| **教训** | **日志会显示任务启动，但不表示任务成功完成；文件修改时间才是真相** |

| 项目 | 内容 |
|------|------|
| **日期** | 2026-02-09 |
| **错误** | 混淆了 2月6日 和 2月8日的任务状态 |
| **教训** | **不同日期的任务状态可能不同，要明确区分日期** |

## 2️⃣ 系统理解

### 教训 #3: PATH 问题

| 项目 | 内容 |
|------|------|
| **日期** | 2026-02-08, 2026-02-09 |
| **场景** | cron 定时任务执行失败 |
| **错误** | cron 任务不加载用户的 PATH 环境变量 |
| **现象** | `line 14/19/26/32: openclaw: command not found` |
| **解决** | 在脚本开头添加: `export PATH="/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:$PATH"` |
| **影响文件** | `~/.openclaw/run-monitor.sh`, `~/.openclaw-daily-backup.sh` |
| **教训** | **cron 环境 ≠ 用户终端环境，必须显式设置 PATH** |

### 教训 #4: 本地操作 ≠ 推送到 GitHub

| 项目 | 内容 |
|------|------|
| **日期** | 2026-02-08 |
| **场景** | 检查 GitHub 备份状态 |
| **错误** | 说"推送到GitHub成功"，但实际上仓库不存在 |
| **后果** | 用户信任受损 |
| **正确做法** | 验证远程仓库是否真的存在: `curl -I https://github.com/user/repo` |
| **教训** | **本地 git commit ≠ 已推送到远程，必须验证远程状态** |

### 教训 #5: 私有仓库 API 返回 404

| 项目 | 内容 |
|------|------|
| **日期** | 2026-02-08 |
| **错误** | 用 GitHub API 检查私有仓库时返回 404，以为仓库不存在 |
| **原因** | 私有仓库需要认证才能通过 API 查看 |
| **正确做法** | 用户告知仓库 URL 后直接信任，或通过 SSH 测试 |
| **教训** | **私有仓库的 API 访问需要认证，不能简单判断为"不存在"** |

### 教训 #12: --no-deliver 只关闭 Telegram，不关闭系统消息

| 项目 | 内容 |

| **错误** | 使用 `--no-deliver` 但任务仍在 main 会话执行 |
